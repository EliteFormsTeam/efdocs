Getting Started:
Overview
There are only two custom html elements you will need to learn in order to become a master at creating robust and performant forms.
The custom <elite-input> tag serves as a direct analog to the native <input> html element. Use <elite-input> to build INDIVIDUAL input fields. These are the building blocks for your form and where you should start!
The custom <elite-form> tag serves as a direct analog to the native <form> html element. Use <elite-form> to wrap all the <elite-input> fields you created. Every <elite-form> comes with a prebuilt with a customizable Submit button. When a user clicks submit, <elite-form> will check each and every input field nested within, and only invoke the developer supplied callback function when no errors exist.
If you require an input type or custom input field that is not already supported by <elite-input>, rest assured knowing that you can add native <input> fields and <elite-form> will include those input values in its values objet. NOTE: <elite-form> can only validate values from <elite-input> fields. Also, keep in mind that ALL input fileds (whether native or elite-forms) requires a class name of 'elite-input'.
Using the <elite-input> tag
Every <elite-input> field MUST contain:
  1. A unique id attribute
  2. A supported type attribute
  3. A class name of 'elite-input'
Use our custom <elite-input> tag with your desired type attribute to create any input field.
Types supported include:
  • Text
    • Email
    • Password
    • Date
  • Radio
  • Checkbox
  • Select
  • Range
  • Textarea
Using the <elite-form> tag
The <elite-form> element can be used as a parent element to bind child elements together yielding a composite form element. The <elite-form> element also renders a submit button at the bottom of the element which when clicked, checks its child elements’ validation status before executing a callback function passed in by the developer.
Every input field nested within an <elite-form> tag must contain:
  1. A callback function. This function will be invoked when the submit button is clicked and no errors exist in the form
NOTE: the callback function must take at least one argument - an object that contains all the values of each input field.

Our custom <elite-form> element allows you to validate the entire form before a submit event and comes equipped with a Submit button.

Field Validation

Include the validationRules attribute in your <elite-input> element with the rules you’d like applied to the form field.

Form Validation

The submit event will not fire successfully unless all fields have passed its validation rules. Our form validator confirms the final state of each element before completing.

Attributes
Attributes for <elite-input>:
  1. id (REQUIRED)
    • Every elite-input MUST have a unique id
  2. type (REQUIRED)
    • Every elite-input MUST include a type attribute
    • Possible type attributes
      • Text
        • Email
        • Password
        • Date
      • Radio
      • Checkbox
      • Select
      • Range
      • Textarea
  3. class (REQUIRED)
    • Every elite-input MUST include a class name "elite-input". You may add additional classes as needed.
  4. label
    • Displays descriptive text above the input field
  5. name
    • Used in certain validation rules to provide a more custom error response
  6. placeholder
    • Displays descriptive text inside input field. Used with text type inputs.
  7. note
    • Displays descriptive text below the input field
  8. validationRules
    • Must be an object with rules you’d like applied to the field. Please see our list of available rules here[insert link].
  9. errorBehavior
    • Determines when error messages render: on every input (default), on every blur event, on a 500 ms debounce
      • Default
        • Not specifying an error behavior results in error messages rendering immediately after each input and persist until there are no errors
      • Debounce
        • Error messages will render when the field detects a pause on input and delay any validation functions from executing until after 500ms have passed.
      • Blur
        • Error messages will render when the field detects a blur event.
  10. conditional
    • Used only when nesting an elite-input inside another. Used by the parent elite-input, and renders the child elite-input only when user input value matches a specific value
    • Learn more about conditional fields here
  11. validationName
    • Used in certain validation rules to provide a more custom error response.
  12. options
    • Used for radio box, checkbox, and select inputs. Accepts an array of objects with two properties: option and value.
      • Option: the string that will be displayed on the form for the user
      • Value:
        • the string collected for internal data collection.
        • used to group radio boxes and checkboxes together.
  13. min
    • Used for range and date type inputs. Specifies a minimum value
  14. max
    • Used for range and date type inputs. Specifies a maximum value
  15. showIndex
    • Used with range type inputs only. Accepts the boolean: true. Displays the minimum and maximum range values at the ends of the slider.
  16. showVal
    • Used with range type inputs only. Accepts the boolean: true. Displays the value of the slider on change.
  17. defaultHidden
    • Used for select inputs. Displays a hidden & disabled attribute as the field’s default value. Accepts a string.
  18. rows
    • Used for textarea inputs. Sets the height of your text area. If not specified, the default will be set to 4.
  19. cols
    • Used for textarea inputs. Sets the width of your text area. If not specified, the default will be set to 50.
  20. showWordCount
    • Used for textarea inputs. Accepts the boolean: true. Displays a text area’s word count.
  21. Style Attributes
    • The elite input field includes default styling that is both simple and generic. Our design philosophy was to style something that could be used by most developers out of the box. However, learn more about implementing even more custom styling here
Attributes for <elite-form>:
  1. onSubmit (REQUIRED)
    • Every elite-form MUST have an onSubmit attribute
    • onSubmit value must be a function definition. This callback will be invoked on a form submission event if all input fields are error free.
  2. buttonName
    • Used to change the name of the submit form button
    • Default value is 'Submit'
  3. badFormMessage
    • Used to change the error message that is displayed if there input fields contain errors and the submit button is clicked.
    • Default value is '!! Missing Fields !!'
  4. Style Attributes
    • The elite form field includes default styling that is both simple and generic. Our design philosophy was to style something that could be used by most developers out of the box. However, learn more about implementing even more custom styling here

after
  • Description: used only with the date input. Checks if the input date is AFTER a specified date. Can be used in conjunction with the min / max attributes, but is not required.
  • Implementation:
    • Input type must be date
    • The validationRule ‘after’ property requires a value of an array as input. The array can accept up to 3 elements, all must be number types and correspond accordingly [ year, month, day ]
      • If only one element is provided: input dates BEFORE provided year will render an error
      • If two elements are provided: input dates BEFORE provided year and month will render an error
      • If three elements are provided: input dates BEFORE provided year, month and day will render an error
alpha
  • Description: checks if input contains ONLY alphabetical characters
  • Implementation:
    • It is encouraged to only use this rule with text inputs. However, it can also be used with other input types depending on use case
    • The validationRule ‘alpha’ property requires a boolean: true value
alphanumeric
  • Description: checks if input contains ONLY alphabetical characters or numbers
  • Implementation:
    • It is encouraged to only use this rule for text inputs. However, it can also be used with other input types depending on use case.
    • The validationRule ‘alphanumeric’ property requires a boolean: true value
before
  • Description: used only with the date input. Checks if the input date is BEFORE a specified date. Can be used in conjunction with the min / max attributes, but is not required.
  • Implementation:
    • Input type must be date
    • The validationRule ‘before’ property requires a value of an array as input. The array can accept up to 3 elements, all must be number types and correspond accordingly [ year, month, day ]
      • If only one element is provided: input dates AFTER provided year will render an error
      • If two elements are provided: input dates AFTER provided year and month will render an error
      • If three elements are provided: input dates AFTER provided year, month and day will render an error
between
  • Description: used with text, range and checkbox inputs.
    • Text: checks if input value’s length is between a specified range
    • Range: checks if input value is within a specified range
    • Checkbox: requires a user to select a specified minimum and maximum number of checkboxes
    NOTE: Between does not validate date inputs. Use the before or after validation rule to handle data validation for date inputs.
  • Implementation:
    • Appropriate input types include text, password, range, textarea, checkbox
    • The validationRule ‘between’ property requires a value of an array with two elements. The first element specifies the minimum value, and the second element specifies the maximum value.
checkExisting
  • Description: Accepts an asynchronous function that makes a call to your database to check if there is an already existing record.
    • Note: Your function must return a boolean: true if your server successfully returns a query for the validation function to throw an error.
  • Implementation:
    • The validationRule ‘checkExisting’ property requires a callback function definition
    • The callback function should make an API request to your database and return a boolean value of TRUE if the input value already exists in your database
email
  • Description: checks if input is a valid email address and renders an error message if the input does not have valid email syntax.
  • Implementation:
    • type attribute must be text: email
    • The validationRule ‘email’ property requires a boolean: true value
endsWith
  • Description: compares the input string against an array of strings to determine if the input has a valid ending.
  • Implementation:
    • type attribute must be text
    • The validationRule ‘endsWith’ property requires an array of strings as its value. Each string represents a valid ending.
matches
  • Description: compares the input string against a custom regex value or an array of strings with your desired values.
  • Implementation:
    • If using an array of valid string values
      • type attribute must be text
      • The validationRule ‘matches’ property requires a value of an object with two properties: type and values
        • type: ‘string’
        • values: an array of strings that you want as valid inputs
    • If using a custom regex string
      • type attribute must be text
      • validationRule ‘matches’ property requires a value of an object with only one property: regex
        • regex: the custom regex string you would like to match
max
  • Description: used with text, range and checkbox input types.
    • Text: checks if input string value is at most a maximum length
    • Range: checks if input range value is less than a maximum number
    • Checkbox: requires user to select at most a maximum number of options
    NOTE: Max does not validate date type attributes. Use the before or after validation rule to handle data validation for date inputs.
  • Implementation:
    • Appropriate type attributes include text, password, range, textarea, checkbox
    • validationRule ‘max’ property requires a Number value.
maxWords
  • Description: checks if input has a maximum word count
  • Implementation:
    • Works with text and textarea inputs.
    • The validationRule ‘maxWords’ property requires a Number value that represents a maximum word count
min
  • Description: used with text, range and checkbox input types.
    • Text: checks if input string value has a minimum length
    • Range: checks if input range value is greater than a minimum number
    • Checkbox: requires user to select a minimum number of options
    NOTE: Min rule does not validate date inputs. Use the before or after validation rule to handle data validation for date inputs.
  • Implementation:
    • Appropriate type attributes include text, password, range, textarea, checkbox
    • validationRule ‘min’ property requires a Number value.
minWords
  • Description: checks if input has a minimum number of words
  • Implementation:
    • Works with text and textarea inputs.
    • The validationRule ‘minWords’ property requires a Number value that represents a minimum word count
matches
  • Description: compares the input string against a custom regex value or an array of strings. An error will occur if a match is found.
  • Implementation:
    • If using an array of forbidden string values
      • type attribute must be text
      • The validationRule ‘not’ property requires a value of an object with two properties: type and values
        • type: ‘string’
        • values: an array of strings that are invalid inputs
    • If using a custom regex string
      • type attribute must be text
      • validationRule ‘not’ property requires a value of an object with only one property: regex
        • regex: the custom regex string for which matching events will render errors
number
  • Description: checks if input contains ONLY numerical characters
  • Implementation:
    • It is encouraged to only use this rule with text inputs. However, it can also be used with other input types depending on use case
    • The validationRule ‘number’ property requires a boolean: true value
reqLower
  • Description: checks if input contains AT LEAST one lowercase alphabetical character. This rule is different from ‘alpha’ as it does not render an error if other character types exist in input value.
  • Implementation:
    • Works with text and password inputs.
    • The validationRule ‘reqLower’ property requires a boolean true value
reqNumber
  • Description: checks if input contains AT LEAST one number. This rule is different from ‘number’ as it does not render an error if other character types exist in input value.
  • Implementation:
    • Works with text and password inputs.
    • The validationRule ‘reqNumber’ property requires a boolean: true value
reqSpecialChar
  • Description: checks if input contains AT LEAST one special character ( ! @ # $ % ^ & * ? )
  • Implementation:
    • Works with text and password inputs.
    • The validationRule ‘reqSpecialChar’ property requires a boolean: true value
required
  • Description: checks to verify the input is not empty. This validation rule can be used with any input type.
  • Implementation:
    • The validationRule ‘required’ property requires a boolean: true value
reqUpper
  • Description: checks if input contains AT LEAST one uppercase alphabetical character. This rule is different from ‘alpha’ as it does not render an error if other character types exist in input value.
  • Implementation:
    • Works with text and password inputs.
    • The validationRule ‘reqUpper’ property requires a boolean: true value
Conditional Fields
What are conditional fields?
Conditional fields allow you to nest an elite-input field inside another elite-input field. When user input on the parent elite-input field matches a certain criteria, the nested elite-input field will appear.
How to use conditional fields?
In the parent elite-input field, use the conditional attribute and assign it the value of an array. The array should contain a string with the valid value that will cause the conditional child elite-input field to render.
Nest a child elite-input field normally. There are no limitations to how the nested elite input field is customized.
Styling Tips
Additional Styling Attributes
Both elite-input and elite-form come with additional attributes that developers can use to add even more customizability
Attributes to style <elite-input>:
  1. styles
    • Used to override the CSS of the most outer div of each elite-input.
  2. labelStyles
    • Used to override the CSS of labels
  3. inputStyles
    • Used to override the CSS of input fields, including all input field type and also textarea and select
  4. noteStyles
    • Used to override the CSS of notes
  5. errorStyles
    • Used to override the CSS of errors
  6. showWordCountStyles
    • For textarea, used to override the CSS of the div, class name ‘show-word-count’, which is displayed when showWordCount is true.
  7. radioCheckboxContainerStyles
    • For checkbox and radio type input, used to override CSS of the div, class name ‘radio-checkbox-container’, wrapping all checkbox/radio inputs and options.
  8. optionStyles
    • For checkbox and radio type input, used to override CSS of the divs, class name ‘options’, wrapping each checkbox/radio input and a corresponding option.
  9. rangeMinmaxIndexboxStyles
    • For range type input, used to override CSS of the div, class name ‘range-minmax-indexbox’, where the minimum and maximum indexes are shown.
  10. rangeValueboxStyles
    • For range type input, used to override CSS of the div, class name ‘range-valuebox’, where the current value is shown.
Attributes to style <elite-form>:
  1. buttonStyles
    • Used to override the CSS of the submit button.
  2. badFormMessageStyles
    • Used to override the CSS of the bad form message, class name ‘sbm-err-msg’ which will be displayed when the submitting is failed.
  3. buttonContainerStyles
    • Used to override the CSS of the div, class name ‘button-container’, which is wrapping the submit button and the bad form message.
